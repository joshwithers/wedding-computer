# Growth-Readiness Audit — wedding.computer

**Date:** 2026-06-10
**Scope:** Full codebase (~52k lines) including the uncommitted sync-engine branch, audited against a target of thousands of new users per year (≈3–5k vendor accounts within 2–3 years, continuous CalDAV/CardDAV/vault-sync traffic).
**Method:** 9 parallel specialized auditors (tenant isolation, D1 scale, Workers platform limits, auth, integration security, sync engine, email pipeline, ops/lifecycle, uncommitted-diff review) → 89 raw findings → 54 after dedup → adversarial verification (one skeptic agent per finding, instructed to refute). 18 findings were fully verified (0 refuted, several severities recalibrated). Verification of the remaining 36 was cut short by an API spend limit; those were recovered from the auditor transcripts and the 11 high-severity ones were manually spot-checked against the code — spot-check results are noted inline.

**Severity counts (post-verification):** 1 critical · 16 high · 28 medium · 8 low.

## CRITICAL

### C1. Storage-sync ingest trusts markdown frontmatter `id` — full cross-tenant takeover ✅ verified · ✔ FIXED (2026-06-29)
**`src/storage/sync.ts:134`** (uncommitted branch — fix before committing)

`applyPulledFile()`'s wedding branch takes the wedding id straight from vendor-supplied frontmatter (`markdownToWedding` → `fm.id`, `src/storage/weddings.ts:125`), then:
- `syncToWeddingsTable` (`sync.ts:593-668`) runs `INSERT INTO weddings ... ON CONFLICT(id) DO UPDATE` — overwriting **any** wedding in the shared D1 by id, with no ownership check.
- `ensureSyncedWeddingMembership` (`sync.ts:670-697`) inserts a `wedding_members` row with `role='vendor', can_manage=1, status='active'` for the calling vendor if not already a member — granting full manager access (couple PII, documents, invoices, member emails via `getWeddingMembers`).

The contacts branch has the same root flaw: `syncToContactsTable` (`src/storage/contacts.ts:662-717`) is `ON CONFLICT(id) DO UPDATE` keyed only on contact id, so a crafted `contacts/x.md` with another vendor's contact id overwrites that contact.

Reachable three ways: vault API `PUT /vault/v1/file/...` (`vault-api.ts:212` — `validatePulledFile` only checks parseability), the GitHub push webhook, and the 5-minute cron. Re-applies on every sweep, so the damage is sticky. Wedding ids leak via URLs, vCard UIDs, MCP exports, and to any vendor ever removed from a wedding. Notably, the **todo branch already does the membership check** (`sync.ts:159-171`) — the wedding and contact branches simply omit the same guard.

**Fix:** Before upserting a wedding, if the row exists, require an active `wedding_members` row for the calling vendor (mirror the todo branch); reject otherwise. Never auto-grant membership from ingested files. Scope the contact upsert with `AND vendor_id = ?` (or verify the existing row's vendor before writing).

**Resolution (2026-06-29):** Both branches now guard. The **wedding branch** rejects via `weddingWriteAccess()` (`sync.ts` — `'new' | 'member' | 'forbidden'`, returns `{ applied: 'ignored', reason: 'wedding belongs to another account' }`) and never auto-grants membership on an existing wedding. The **contact branch** now does the same ownership check before writing — `SELECT vendor_id FROM contacts WHERE id = ?`; if the id already belongs to a different vendor it returns `{ applied: 'ignored', reason: 'contact belongs to another account' }` and writes nothing (no `contacts` clobber, no leaked `file_index` row). Defence in depth at the SQL layer was already present: `syncToContactsTable` upserts with `ON CONFLICT(id) DO UPDATE SET ... WHERE contacts.vendor_id = excluded.vendor_id` (shipped with the original vault-sync commit), so the cross-tenant data write was already a no-op — the app-layer guard closes the remaining no-op-write/index-pollution gap and matches the wedding branch. Covered by `applyPulledFile — cross-tenant wedding guard (C1)` and `… cross-tenant contact guard (C1)` in `src/storage/sync-engine.test.ts`.

## HIGH

### H1. MCP by-id wedding tools skip membership checks — cross-tenant PII read ✅ verified
**`src/routes/mcp.tsx`** (live code; path renamed since the original audit)
`get_wedding` (`SELECT * FROM weddings WHERE id = ?`), `get_wedding_log` (unscoped `wedding_log` export, `db/wedding-log.ts:46`), and `get_wedding_credits` (member emails + names via `getWeddingMembers`) take a caller-supplied wedding id with no membership gate — while sibling tools `list_weddings`/`get_wedding_todo` are correctly scoped. A vendor removed from a wedding keeps reading it via MCP, defeating the removal control.
**Fix:** membership JOIN gate on every by-id tool; return "not found" on failure.

### H2. 5-minute sync cron iterates every git vendor serially, unbounded ✅ verified
**`src/services/storage-sync.ts:47`**
No LIMIT, no cursor, no time budget, no queue fan-out. ~2 full recursive GitHub Trees fetches per idle vendor, up to ~150 GitHub calls per active vendor (25 weddings × 3 files, each write = GET+PUT). The 1,000-subrequest cap (D1/KV calls count too) is breached at roughly **125–150 idle git vendors**; the loop throws partway, the same head-of-list vendors always win, tail vendors silently stop syncing, and `storage-sync.ts:76-79` swallows the failures.
**Fix:** one queue message per vendor (or persisted cursor + per-run budget), process least-recently-synced first, track `last_synced_at`.

### H3. Daily digest + payment reminders: all vendors in one invocation ✅ verified
**`src/services/notifications.ts:696`**
4 D1 queries per vendor, then a synchronous Resend fetch per email (bypassing the provisioned EMAIL_QUEUE), all inside one `0 20 * * *` invocation. Dies around **150–250 vendors**. Because `sendPaymentReminders` runs after the digest in the same handler, once the budget is exhausted **payment reminders fail for every vendor that day** — your cash-flow feature silently stops.
**Fix:** enqueue one job per vendor; filter opted-out vendors in SQL; collapse the per-vendor queries.

### H4. Contact filename dedup reads only the first 1,000 R2 objects — silent overwrite ✅ verified
**`src/storage/contacts.ts:646`**
`listExistingFilenames` never follows the R2 cursor (cap 1,000). Past 1,000 contact files, a new enquiry from a common name (second `john-smith.md`) overwrites the existing file and `upsertIndex`'s `ON CONFLICT(vendor_id, file_path)` repoints the index row to the new contact — destroying the old one in the lead-capture path (`form.tsx:309`). Legacy D1 contact row survives today, but that table is slated for removal.
**Fix:** don't list at all — probe `file_index` (`SELECT 1 WHERE vendor_id=? AND file_path=?`, unique index already exists) and suffix on collision.

### H5. Email queue: no DLQ, no Resend throttle, drops mail ✅ verified
**`wrangler.toml:35`**
`max_retries=3`, no `dead_letter_queue`, no `max_concurrency`. Broadcasts fan out per-recipient and the consumer fires ~10 Resend calls instantly per batch — well past Resend's ~2 req/s. Worse than claimed: `deliver()` (`notifications.ts:71-107`) never throws, so failed new-lead/invoice notifications are **acked after a single attempt — zero retries** (a D1 `emails` row gets `status='failed'` but nothing surfaces or re-drives it).
**Fix:** DLQ binding; `max_concurrency=1` or pacing for Resend-bound types; let transient failures throw so retries happen; distinguish permanent failures.

### H6. `/dev/login/:email` auth bypass gated only on absence of the `cf-ray` header ✅ verified
**`src/routes/auth.tsx:128`**
Creates a session for any email (including `is_admin` accounts) and even creates new users, bypassing the invite gate. Not exploitable behind Cloudflare's edge today, but this is a one-condition fence around total account takeover for any non-production or misconfigured deployment — and the cookie is set without `secure: true`.
**Fix:** gate on an explicit dev-only env binding, or strip the route from production builds.

### H7. Sync pull silently overwrites newer D1 edits with stale file content 🔎 spot-checked
**`src/storage/sync.ts:256`** (uncommitted branch)
The pull path compares file etag vs `file_index` only; when they differ it applies file content into D1 with no comparison against D1's `updated_at` and no conflict record. An in-app edit made between file change and sync is silently lost.

### H8. Empty/truncated storage listing mass-deletes the file index 🔎 spot-checked
**`src/storage/sync.ts:378`** + **`src/storage/github.ts:252`** (✅ truncation half verified)
Pass 3 deletes every `file_index` row whose path is missing from the current listing. GitHub Trees `truncated: true` is only logged (`github.ts:252-254`), so a large vault gets its missing files treated as deletions. Any backend bug returning a partial/empty list cascades the same way.
**Fix:** abort sync on truncation; treat a listing that would delete more than N% of the index as an error requiring confirmation.

### H9. No per-vendor serialization across three concurrent sync writers 🔎 spot-checked
**`src/services/storage-sync.ts:69`** (uncommitted branch)
Cron, GitHub webhook, and vault PUT can all run `syncVendorStorage` for the same vendor concurrently — no lock, mutex, or in-progress flag anywhere. Concurrent GitHub Contents-API pushes 409; interleaved pulls/pushes can double-apply or duplicate `wedding.md` on folder renames.
**Fix:** per-vendor KV lock with TTL (or a Durable Object per vendor) around sync; debounce webhook deliveries.

### H10. Vendor compose endpoint: unthrottled outbound email from @wedding.computer 🔎 spot-checked
**`src/routes/vendor/emails.tsx:223`**
`POST /app/emails/send` sends arbitrary email to arbitrary recipients with no rate limit or quota (no blanket rate limit exists on `/app` routes). A free signup is an open spam relay that can poison the shared sending domain that magic-link auth depends on.
**Fix:** per-vendor daily quota + per-minute rate limit; consider verified-recipient constraints for new accounts.

### H11. No bounce or complaint suppression anywhere 🔎 spot-checked (grep: zero hits)
Hard-bouncing addresses are re-emailed daily by digests/reminders. At thousands of sends/day this erodes domain reputation until magic-link delivery (your only login path) suffers.
**Fix:** subscribe to Resend bounce/complaint webhooks, maintain a suppression list, check it in `sendEmailMessage`.

### H12. Broadcast batches by message count only — 256KB batch limit + duplicate sends 🔎 spot-checked
**`src/routes/admin.tsx:751`**
Full rendered HTML is embedded in each queue message and `sendBatch` chunks by 100 messages regardless of size. Queues caps a batch at 256KB total; 100 × several-KB HTML exceeds it → partial enqueue, and re-submitting the form re-sends to everyone already enqueued.
**Fix:** store the body once (D1/R2/KV) and enqueue `{recipient, broadcastId}`; chunk by byte size.

### H13. Account deletion orphans nearly everything 🔎 spot-checked
**`src/routes/account.tsx:761`**
Deletion removes the avatar, the D1 user (cascades), and the current session. It leaves: all vault/document R2 objects, KV secrets (GitHub PAT, webhook secrets, tokens), the GitHub webhook on the vendor's repo, Stripe references, and other active sessions in KV. GDPR exposure plus unbounded storage cost.

### H14. `weddings.created_by_user_id` has no ON DELETE action 🔎 spot-checked (schema confirmed)
**`schema.sql:121`**
`REFERENCES users(id)` with no action — deleting any user who created a wedding either fails the FK (breaking account deletion) or orphans the wedding tree, depending on enforcement. Test account deletion for a wedding-creating vendor.

### H15. schema.sql is missing tables that migrations added and code queries ✅ confirmed by direct diff
**`schema.sql`**
`forms` and `form_submissions` (migration 031) are absent from schema.sql, which is exactly the file `db:migrate:remote`/`db:migrate:local` executes. A fresh environment (new dev machine, staging, internal deployment, disaster recovery) gets a database the code crashes on. There's also no migration-tracking table — applying migrations to prod is manual and unordered.
**Fix:** switch to `wrangler d1 migrations apply` (it tracks applied migrations), and regenerate schema.sql from the real database (or delete it as a source of truth).

### H16. One sync token spans vault writes, CardDAV PII, and MCP — and lives in URLs ⚠️ recovered, not verified
**`src/routes/vendor/settings.tsx:727`**
The same token authenticates read-write vault sync, CardDAV (all client PII), CalDAV, and MCP, and is embedded in DAV/feed URLs that get pasted into third-party clients and logged by proxies. One leak = full read-write access to the vendor's business. Consider per-surface tokens and per-token revocation.

## MEDIUM

### Platform & D1 scale
- **M1. Rate limiting is non-atomic KV read-then-write** ✅ verified — `src/middleware/rate-limit.ts:42`. Concurrent bursts bypass all limits (login 5/min, public forms 10/min); the per-request KV put on the vault limiter also hits KV's ~1 write/sec/key and can 500 legitimate sync clients. Move to a Durable Object or D1 atomic upsert; stop writing on every allowed request.
- **M2. Nightly busyness aggregation builds a contacts × events cross-product per date** ✅ verified (downgraded from high) — `src/db/busyness.ts:104`. Two one-to-many joins on the same date key explode on popular Saturdays; no index on `contacts.wedding_date`; per-location counts double-attributed. Split into two GROUP BY queries merged in JS; add the index.
- **M3. `emails` table stores full HTML of every system notification, never pruned, never read** ✅ verified — `src/services/email.ts:64`. Write-only unbounded growth against D1's 10GB hard cap (~1–2 years runway at full scale; all-tenant write failure when hit). Stop storing `body_html` for `is_system` sends; add retention purge.
- **M4. Append-only tables have no pruning path** ✅ verified — sessions (expired rows never deleted), `audit_log` (one row per vault PUT), `analytics_events`, `wedding_log`, `contact_activities`, `import_jobs.raw_data` (full CSV stored twice, plus per-row duplication in `import_records`). Add batched retention deletes to the nightly cron.
- **M5. GitHub `list()` fetches the entire recursive repo tree; every sync fetches it twice** ✅ verified — `src/storage/github.ts:237`, `sync.ts:87-91`, `vault-api.ts:113-124`. Compare branch-head SHA against last-synced SHA to skip unchanged vaults; share one tree per sync cycle.
- **M6. Every GitHub write is GET-then-PUT, and our own pushes re-trigger our webhook → full rescan** ✅ verified — `github.ts:134`, `webhooks.ts:63-79`. Steady state is bounded by etag no-ops, but bulk activity (initial vault sync) can hit GitHub's ~80/min content-creation secondary limit. Pass the cached `file_index` etag as the PUT sha; skip webhook syncs for the app's own commits (recognizable committer).
- **M7. CalDAV serves the vendor's entire event history on every poll** ✅ verified — `src/routes/caldav.ts:199`. Unbounded `SELECT *` per PROPFIND, COUNT+MAX ctag scan per poll, full enriched REPORT on any change, per multiple devices polling every 5–15 min. Bound the window (match `feed.ts`), cache the ctag, add RFC 6578 sync-collection later.
- **M8. GitHub binary write base64-encodes byte-by-byte in a JS string loop** ✅ verified (downgraded from high) — `github.ts:177` with a 10MB buffered file from `files.tsx:176`. Real but interactive-path-only; chunk the encoding.
- **M9. 903KB `noim-blank.pdf` bundled into the Worker** ⚠️ recovered — `wrangler.toml:47` Data rule inflates every isolate. Serve from R2/assets instead.
- **M10. `new_lead` queue handler reads contact markdown from GitHub at notify time** ⚠️ recovered — `src/index.tsx:513`. Storage outage = dropped lead notifications (compounded by H5's no-retry `deliver()`). Embed needed fields in the queue message.

### Security & auth
- **M11. Session validation is KV-only; D1 `sessions` rows are raw tokens, never consulted** 🔎 spot-checked — `src/middleware/auth.ts:13` (`resolveSession(c.env.KV, ...)`), `src/db/sessions.ts:9`. You can't revoke sessions by clearing D1; the table is a live-credential dump if the DB ever leaks. Hash the stored token; decide one source of truth.
- **M12. Magic link consumed by a plain GET; single-use relies on KV get-then-delete** ⚠️ recovered — `src/services/auth.ts:39`. Email security scanners follow links (consuming or logging in), and KV eventual consistency leaves a replay window. Use an interstitial POST to consume; consider D1 for the one-time-use mark.
- **M13. Email/account enumeration oracles** ⚠️ recovered — invite-gate response divergence and passkey options leaking credential IDs (`src/routes/auth.tsx:63`). Normalize responses.
- **M14. MCP endpoint has no rate limit and is outside the shared auth-failure throttling** ⚠️ recovered — `src/routes/mcp.tsx` (path renamed since the original audit).
- **M15. Vendor secrets (GitHub PAT, API keys) lack application-layer encryption; plaintext tokens still accepted in D1 `storage_config`** ⚠️ recovered — `src/services/secrets.ts:21`. KV storage is fine, but legacy plaintext paths should be migrated and rejected.
- **M16. Inbound email: no SPF/DKIM/sender verification, no size cap, no rate limit, stored in D1** ⚠️ recovered — `src/services/inbound-email.ts:57`. Spoofed mail gets attributed to real CRM contacts; unauthenticated D1 growth vector.
- **M17. Vault API rate limit is per-IP only** ⚠️ recovered — `src/routes/vault-api.ts:37`. NAT'd users get throttled together; a single vendor can spread across IPs. Key on token/vendor.

### Sync engine (uncommitted branch)
- **M18. `file_conflicts` CHECK constraint excludes 'todo'/'log' entity types** ⚠️ recovered — `schema.sql:541` vs code that now syncs todo/log files. Conflict insert for those types will throw.
- **M19. `file_index` lacks UNIQUE on (vendor_id, entity_type, entity_id)** ⚠️ recovered — `migrations/038:16`. Single-row lookups assume uniqueness; folder renames can produce two rows for one entity.
- **M20. Every vault PUT to a git backend self-triggers a full webhook resync** ✅ verified (as part of M6) — `vault-api.ts:202`. Write amplification: each Obsidian save = commit + webhook + 2 tree fetches.

### Email pipeline
- **M21. Queue retries are non-idempotent** ⚠️ recovered — `src/services/email.ts:95`. Each retry inserts a new `emails` row and re-sends with no Resend `Idempotency-Key` — duplicate emails on transient failures (where retries happen at all; see H5).
- **M22. No RFC 8058 List-Unsubscribe headers on broadcast/waitlist mail** ⚠️ recovered — `src/index.tsx:713`. Gmail/Yahoo bulk-sender requirements kick in at exactly your target volumes.

### Ops
- **M23. No tests on Stripe webhook/payment or magic-link/session paths** ⚠️ recovered — strong storage-sync test coverage, nothing on money or login.
- **M24. Observability can't answer "which vendor's sync is failing"** ⚠️ recovered — unstructured `console.log`, no per-vendor correlation, no alerting on cron/queue/webhook failure. At 5k vendors, silent partial failure (H2/H3) is invisible.
- **M25. Single-copy R2 with no versioning is the blast radius of a bad sync or deploy** ⚠️ recovered — `wrangler.toml:26`. Combined with H8 (mass index deletion) and D1 Time Travel's ~30-day window, a bad sync-engine bug has no recovery story for file content. Enable R2 object versioning or periodic snapshot-to-second-bucket.
